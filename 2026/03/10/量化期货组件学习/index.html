    量化期货组件学习

    发表于 2026-03-10 分类于 量化

    在高性能交易引擎中，通常有三个核心组件分别为行情/alpha和策略，本文学习一下他们的基本概念。

    定义

    1. 行情 (Market Data / MD)

    • 本质： 交易所状态的离散采样。
    • 底层形式： 在期货中通常是 CTP 的 OnRtnDepthMarketData 或更底层的二进制流（如 SBE 编码）。它不仅包含价格，还包含**订单簿（LOB）**的静态快照。
    • 职责： 提供一切计算的原始信源（Raw Input）。

    2. Alpha (信号 / Signal)

    • 本质： 一种从噪声中提取的概率函数。
    • 逻辑表达： $Alpha = f(MD_{current}, MD_{history})$。
    • 职责： 它的唯一目标是输出一个方向（看多/看空）和一个强度（信心值）。它不考虑你现在手里有多少钱，也不考虑单子能不能成交。

    3. 策略 (Strategy / Logic)

    • 本质： 一个带状态的决策引擎。
    • 逻辑表达： $Action = g(Alpha, MD_{LOB}, Position, Risk_Limits)$。
    • 职责： 负责执行（Execution）。它读行情是为了看“买一卖一”的价格和深度，从而决定是直接吃单（Aggressive）还是挂单等待（Passive）。

    交互逻辑

    它们之间通过事件驱动 (Event-Driven) 进行交互。

    交互点 触发动作 交互内容
    信号触发 行情更新 (Tick/SBE) 行情机推给 Alpha，Alpha 重新计算出信号值并推给策略。
    风险校验 行情剧烈波动 策略感知行情变化，即使 Alpha 没变，策略也可能主动撤单或止损。
    成本控制 订单簿深度变化 策略读行情 LOB，发现滑点过大，修改报单价格或撤单。
    反馈循环 报单成交 (Trade) 策略通知 Alpha：“我的单子成交了，现在的市场冲击可能影响你的下一个信号。”

    期货量化的一天：全流程硬核拆解

    既然你正在处理 SBE 协议和 CTP 接口，我们可以把这套流程对应到你的开发逻辑中。

    1. 盘前：初始化与状态一致性 (Pre-Market)

    • 加载配置： 解析 YAML 配置文件（如你正在写的告警规则或交易参数）。
    • 系统自检： 检查网络延迟，确保到交易所的网线是通的。
    • 状态恢复： 读入昨天的结算单，恢复持仓状态。量化系统必须知道自己手里有多少“货”，防止程序重启后乱下单。
    • 静态数据同步： 获取当天的合约列表、手续费率、涨跌停板限制。

    2. 盘中：低延迟执行循环 (Intra-Market)

    这是 C++ 发挥威力的地方，核心是一个极速循环：

    1. 收包： 网卡收到 UDP 行情包，通过 SBE 快速解析成内存结构。
    2. 分发： 行情同时甩给 Alpha 模块和策略模块。
    3. 计算： Alpha 算出方向；策略结合 Alpha 和当前订单簿深度，判断是否符合下单条件。
    4. 报单： 调用 ReqOrderInsert 发往交易所。
    5. 监控： 你的 Python 监控脚本实时解析日志，如果发现 ImportError 或连接断开，立即发出告警。

    3. 盘后：数据闭环与进化 (Post-Market)

    • 数据落库： 将全天的原始二进制行情转存为高效的压缩格式。
    • 对账 (Reconciliation)： 检查本地成交记录和柜台返回的结算单是否一分钱不差。
    • 重放 (Replay)： * Debug 重放： 如果盘中某笔单子下错了，用 GDB 挂载 Replay 程序，喂入当天的行情包，看代码逻辑在哪个 if 分支走歪了。
      • 参数优化： “如果我今天把 Alpha 的阈值调高 10%，利润会更好吗？”在 Replay 环境中跑模拟，这是进化的唯一途径。
