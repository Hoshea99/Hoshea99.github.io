Linux调试经验升级版

发表于 2025-12-08 分类于 linux

这是一份高性能后台开发/性能调优的命令指南，附有真实案例。

分成了CPU调优、内存调优以及经典的perf+valgrind+FrameGragh三驾马车三大部分。包含了调试思路和具体案例。

内存

free

首先使用free -h命令粗调试，看看到底是不是因为内存导致的程序卡死。

  • Mem栏中，free并不是可用内存，他低并不代表内存不够，因为linux会用空闲内存去支持文件IO。需要看的是Available。这一栏低了才是代表内存不足。
  • 重点看Swap中的used，used代表出现了内存交换，只要这个值大于0，则代表内存出现了不足现象。

vmstat

如果已经看出内存不足，现在则使用vmstat 1指令去确认问题出现在哪个部分：

这里有几个指标：

  • r：等待运行进程，表示现在在等待CPU调度，这个值大于CPU核数则说明CPU资源不足
  • b：不可中断睡眠，表示该进程正在等待IO
  • memory和swap变化不大，说明内存正常
  • wa：等待磁盘IO，增长到30%说明磁盘IO占据时间太长
  • us：说明用户态CPU消耗过高
  • sy：系统调用占比，说明锁耗时或者有过多小内存分配问题

通过perf观察是系统调用高（而不是计算过高）后，来这里看一下，如果是wa高，说明是IO耗时，如果是sy高，说明是系统调用过多（比如锁，或者过多小内存分配）。

比如这个例子中，read调用过多，考虑是不是缓冲区太小导致。

关键观察点：

  • 大量的 read() / write() / sendmsg() / recvmsg()： 可能是由于网络或文件 I/O 过于频繁或缓冲区过小，导致需要频繁进出内核。
  • 大量的 futex() / clone() / sched_yield()： 强烈暗示锁竞争或调度问题。futex 是 Linux 上实现锁和同步原语（如互斥锁、条件变量）的基础。

pmap

查看具体原因，使用pmap -x <PID>进行排查。

如上图所示，anon块为堆上申请的内存，而stack为栈上占据的内存，首先排查栈上申请过多内存（每个栈只有8Kb）。

其次，观察总RSS为2.1G，而大多数内存都是Anon（匿名内存，堆上）获取的，合理推断应用程序频繁创建1MB左右的缓存区。

结论：

  1. 内存泄漏，忘记释放这堆内存
  2. 被释放了，但是mmap特性导致了没有立即释放回操作系统。因此考虑使用内存池，只要一开始创建有限个较大内存，后续都往内存池申请内存。

CPU

CPU的瓶颈一般分为两种：“整机负载”与“单核瓶颈”，“自愿切换”与“被动抢占”。

top

最简单的指令，用于看当前进程的us，sy，wa的值。观察到底是us太高，用户态代码有问题，还是si/so太高，内存不足。

  • us高是好事，说明CPU主要是用于计算，但是太高就说明代码有问题。
  • sy高说明系统调用过多，比如锁、IO等
  • wa高说明等待IO过久

但是top的平均特性会掩盖单核爆了的现象。

mpstat

使用mpstat -P ALL 1观察是否是单核爆炸了。

pidstat

如果每个CPU都比较高，后者解决了单核负载问题后还是卡顿，则需要使用pidstat -w -t去看上下文切换量，排查是不是这个原因导致的。

现象pidstat -w -t 指标根源推断
等待资源cswch/s (自愿切换) 高线程主动放弃 CPU，等待 锁释放 或 I/O 完成。
调度争抢nvcswch/s (非自愿切换) 高CPU 资源稀缺，线程被强制打断。

如果是**cswch/s (自愿切换) 高**，往往是因为锁或者等待IO，需要用perf看看是不是锁的原因，如果是非自愿切换高，则代表是时间片用完了，需要用perf看看热点函数在哪一个消耗了太多计算资源。

perf在后文讲解

高阶工具篇（三驾马车）

即perf、valgrind和FrameGraph

perf

关注点：

  • IPC (Instructions Per Cycle)：指令/时钟周期。如果 < 1，说明大量时间在空转等待（Memory Bound 或 Pipeline Stall）。
  • cache-misses：缓存未命中（结构体未对齐、伪共享）。
  • branch-misses：分支预测失败（大量的 if-else）。

perf record & report：

  • 生成函数热点图。如果 kernel 态占比过高，检查是否 buffer 太小导致 syscall 过于频繁。

解决：

如果IPC过低，则说明了有可能是存在缓存未命中或者分支预测失败，而不是计算量过大

如果缓存未命中，可以考虑优化结构体，减少padding

如果是分支预测失败比较多，则需要减少过多if else或者switch，或者优化这些分支的顺序。

热点图后续在介绍火焰图重点叙述。

火焰图

perf record -g生成的文件，生成perf.data，可以使用火焰图进行分析

  • 看图技巧：

    • X 轴：样本总量（宽度 = 耗时占比）。
    • Y 轴：调用栈深度。
    • 瓶颈特征：“平顶山”（Flat top）。若某函数顶层很宽，说明它消耗了大量 CPU。

  • 案例：发现 JSON 解析函数呈现“平顶山”，优化方案是手写/更换更快的解析器（如 simdjson）。

    函数Total %Self %User % (占 Total %)System % (占 Total %)
    do_work()60%5%5%95%

    说明自身的耗时不多，但是调用的子函数耗时较多，而且System占的多，说明可能是函数内的IO较频繁。
